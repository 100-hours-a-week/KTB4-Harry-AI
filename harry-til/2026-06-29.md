# 📝 TIL (Today I Learned)

## [핵심 배운 내용]

- LangGraph는 상태를 공유하는 노드와 엣지를 그래프로 연결해, 장시간 실행되는 워크플로와 에이전트를 설계하는 저수준 오케스트레이션 프레임워크다.
- Workflow는 개발자가 미리 정한 실행 경로를 따르고, Agent는 주어진 범위 안에서 LLM이 상태를 바탕으로 다음 행동과 도구 사용을 결정한다.
- LangGraph의 기본 구성 요소는 `State`, `Node`, `Edge`이며, `StateGraph`로 이들을 조합한 뒤 `compile()`하여 실행 가능한 그래프를 만든다.
- Tool Calling은 LLM이 호출할 도구와 인자를 결정하는 단계이고, `ToolNode`는 그 요청을 실제로 실행하는 단계다.
- Routing, Parallelization, Loop, `Send`, `Command`를 활용하면 조건 분기, 병렬 처리, 반복 실행과 같은 복잡한 제어 흐름을 표현할 수 있다.

## [개념 정리]

### LangGraph와 AI Agent

#### LangGraph

LangGraph는 **상태를 가진 LLM 애플리케이션의 워크플로와 에이전트를 그래프 구조로 설계하고 실행하는 프레임워크**다.

실행 순서를 LLM에 모두 맡기는 대신 개발자가 그래프를 통해 허용된 흐름을 명시할 수 있다. 이를 통해 분기, 반복, 상태 저장, 중단 후 재개, 사람의 개입이 필요한 작업을 더 안정적으로 구현할 수 있다.

#### Workflow와 Agent

| 구분 | Workflow | Agent |
| --- | --- | --- |
| 실행 경로 | 개발자가 미리 정의 | LLM이 현재 상태에 따라 동적으로 결정 |
| LLM의 역할 | 노드 내부의 처리 수단 중 하나 | 다음 행동과 도구 사용을 판단하는 핵심 주체 |
| 장점 | 예측, 비용 통제, 디버깅이 쉬움 | 실행 경로를 고정하기 어려운 문제에 유연하게 대응 |

Agent도 완전히 자유롭게 동작하는 것은 아니다. 개발자가 제공한 도구와 지침, 그래프의 범위 안에서 상태를 읽고 다음 행동을 선택한다.

### Graph API의 기본 구조

Graph API는 상태 스키마, 노드, 엣지를 명시적으로 선언해 그래프를 만드는 LangGraph의 기본 프로그래밍 인터페이스다.

#### StateGraph와 State Schema

- `StateGraph`: 상태 스키마를 기반으로 노드와 엣지를 등록하는 그래프 빌더 클래스
- State Schema: 그래프의 모든 노드가 공유하는 상태의 키와 타입, 병합 규칙을 정의한 구조
- 상태 스키마는 `TypedDict`, `dataclass`, Pydantic 모델 등으로 정의할 수 있다.

상태 구조를 미리 고정하면 노드가 많아져도 데이터의 흐름을 예측하기 쉽고, 상태 관련 오류를 줄일 수 있다.

#### Reducer와 MessagesState

Reducer는 특정 상태 키에 새 값이 들어왔을 때 **기존 값과 어떻게 병합할지 정의하는 함수**다.

```python
from typing import Annotated

class State(TypedDict):
    results: Annotated[list[str], reducer_function]
```

여러 노드가 같은 상태 키를 갱신하는 병렬 실행에서는 값이 덮어써지거나 충돌하지 않도록 적절한 Reducer가 필요하다.

`MessagesState`는 `messages` 키와 메시지 병합용 `add_messages` Reducer를 미리 제공하는 기본 상태 스키마다. 채팅봇이나 에이전트에서 대화 기록을 직접 병합하는 수고를 줄여 준다.

#### Node와 Edge

- **Node**: 현재 State를 입력받아 작업을 수행하고 상태 업데이트를 반환하는 실행 단위
- **Edge**: 한 Node의 실행이 끝난 뒤 다음에 실행할 Node 또는 종료 지점을 지정하는 연결
- **Routing Function**: 현재 상태를 기준으로 다음 Node를 선택하는 함수로, 그 자체는 Node가 아니다.

#### START, END, Compile

- `START`: 그래프의 입력이 처음 전달될 Node를 지정하기 위한 가상 시작 노드
- `END`: 이후 실행할 작업이 없음을 나타내는 가상 종료 노드
- `compile()`: 선언한 노드와 엣지를 검증하고 호출 가능한 그래프로 변환하는 단계

```python
from langgraph.graph import END, START, StateGraph

builder = StateGraph(State)
builder.add_node("process", process_node)
builder.add_edge(START, "process")
builder.add_edge("process", END)

graph = builder.compile()
result = graph.invoke(initial_state)
```

컴파일된 그래프는 `graph.get_graph().draw_mermaid_png()`와 같은 기능으로 시각화해 실행 구조를 확인할 수 있다.

### Tool Calling과 ToolNode

#### Tool Calling

Tool Calling은 LLM이 사용자 입력을 분석해 **호출할 도구의 이름과 인자를 구조화된 형태로 결정하는 기능**이다. 이 단계에서는 호출 요청만 만들며 도구를 직접 실행하지 않는다.

일반적인 사용 흐름은 다음과 같다.

1. `@tool`로 Python 함수를 도구로 정의한다.
2. `llm.bind_tools(tools)`로 LLM에 사용 가능한 도구를 알려 준다.
3. LLM을 호출한다.
4. 반환된 `AIMessage.tool_calls`에서 도구 호출 요청을 확인한다.

#### ToolNode

`ToolNode`는 LLM이 생성한 `tool_calls`를 받아 실제 도구를 실행하고, 결과를 `ToolMessage`로 만들어 State에 반영하는 사전 구성 노드다.

```python
from langgraph.prebuilt import ToolNode

tool_node = ToolNode(tools)
builder.add_node("tools", tool_node)
```

즉, **LLM은 어떤 도구를 사용할지 판단하고, ToolNode는 그 판단을 실행한다.**

### LangGraph의 제어 흐름

#### Routing

Routing은 입력이나 현재 State를 기준으로 다음 실행 노드를 선택하는 분기 패턴이다. 라우팅 함수를 `add_conditional_edges()`에 연결해 조건부 흐름을 구성한다.

```python
builder.add_conditional_edges("agent", route_fn, {
    "tools": "tools",
    "end": END,
})
```

#### Parallelization

서로 의존하지 않는 여러 노드를 동시에 실행하고, 모든 결과가 준비되면 다음 노드에서 합치는 패턴이다. 처리 시간을 단축하거나 동일한 작업을 여러 관점에서 수행할 때 유용하다.

병렬 노드가 같은 상태 키에 값을 기록한다면 Reducer를 통해 병합 방법을 정의해야 한다.

#### Send

`Send`는 런타임에 만들어진 데이터 목록을 기준으로 같은 노드를 여러 번 실행하는 동적 분기 객체다. 작업 개수를 미리 알 수 없는 map-reduce 형태의 병렬 처리에 적합하다.

```python
from langgraph.types import Send

def assign_workers(state):
    return [Send("worker", {"item": item}) for item in state["items"]]
```

#### Command

`Command`는 하나의 노드에서 **상태 업데이트와 다음 노드 지정을 함께 반환**할 때 사용하는 객체다.

```python
from typing import Literal
from langgraph.types import Command

def node(state) -> Command[Literal["next_node"]]:
    return Command(
        update={"status": "done"},
        goto="next_node",
    )
```

상태를 갱신하면서 동적으로 이동해야 할 때는 `Command`가 적합하고, 상태 갱신 없이 경로만 선택한다면 조건부 엣지가 더 단순하다.

#### Loop와 종료 설계

Loop는 엣지가 이전 노드를 다시 가리키도록 연결해, 종료 조건이 충족될 때까지 같은 흐름을 반복하는 구조다. 도구 실행 결과를 State에 누적한 뒤 LLM이 다시 판단하는 에이전트에 자주 사용된다.

반복에는 두 가지 종료 장치가 필요하다.

- **정상 종료**: 라우팅 조건을 만족하면 `END`로 이동
- **강제 종료**: 최대 반복 횟수를 제한해 무한 루프와 과도한 비용을 방지

강제 한도를 초과하면 정상 결과 대신 `GraphRecursionError`가 발생할 수 있으므로, 가능한 경우 정상 종료 조건을 명확히 설계해야 한다.

### ReAct Pattern

ReAct는 LLM이 추론하고 필요한 도구를 실행한 뒤, 그 결과를 다시 판단에 반영하는 과정을 반복하는 에이전트 실행 패턴이다.

```text
LLM 판단 -> 도구 선택 -> 도구 실행 -> 결과 관찰 -> LLM 재판단 -> ... -> 종료
```

LangGraph에서는 일반적으로 Agent Node, Tool Node, 조건부 엣지, 순환 엣지를 연결해 이 구조를 만든다.

### Functional API

Functional API는 일반 Python 함수에 `@entrypoint`와 `@task` 데코레이터를 적용해 함수 호출 흐름을 LangGraph 워크플로로 실행하는 인터페이스다.

- `@task`: 개별 작업 단위를 정의
- `@entrypoint`: 워크플로의 진입점을 정의

그래프 구조를 명시적으로 조립하는 Graph API와 달리, 기존 Python 함수의 흐름을 유지하면서 LangGraph의 실행 기능을 적용하고 싶을 때 유용하다.
