# 📝 TIL (Today I Learned)

## [핵심 배운 내용]

- LangGraph의 Persistence는 그래프의 State를 Checkpoint로 저장하여 대화의 연속성, 중단 후 재개, 장애 복구와 디버깅을 지원한다.
- Checkpointer는 Persistence를 구현하는 저장 인터페이스이며, `thread_id`를 기준으로 서로 다른 실행 흐름의 Checkpoint를 저장하고 조회한다.
- Thread는 여러 번의 그래프 실행을 하나의 흐름으로 묶는 세션 단위이고, Checkpoint는 특정 시점의 State와 실행 정보를 담은 스냅숏이다.
- Durable Execution은 저장된 Checkpoint를 바탕으로 중단된 장기 실행 작업을 처음부터 다시 수행하지 않고 이어 갈 수 있게 하는 실행 방식이다.

## [개념 정리]

### Persistence 구조

LangGraph의 영속성은 다음 관계로 이해할 수 있다.

```text
Persistence
└── Checkpointer
    └── Thread (thread_id로 구분되는 실행 흐름)
        └── Checkpoint (각 시점의 State 스냅숏)
```

Persistence가 **State를 실행 이후에도 유지하는 기능**이라면, Checkpointer는 그 기능이 실제로 동작하도록 State를 저장하고 불러오는 구성 요소다.

### Persistence

Persistence는 LangGraph가 그래프의 State를 Checkpoint로 저장하고, 이후 같은 실행 흐름에서 다시 사용할 수 있게 하는 기능이다.

이를 통해 다음 기능을 구현할 수 있다.

- 여러 호출에 걸친 대화 상태 유지
- 실행 중단 후 재개
- 실패한 작업의 복구
- Human-in-the-loop
- 과거 상태 조회와 디버깅
- 이전 Checkpoint부터 다시 실행하는 Time Travel

Persistence를 사용하려면 그래프를 컴파일할 때 Checkpointer를 연결하고, 실행할 때 `thread_id`를 전달한다.

```python
from langgraph.checkpoint.memory import InMemorySaver

checkpointer = InMemorySaver()
graph = builder.compile(checkpointer=checkpointer)

config = {
    "configurable": {
        "thread_id": "conversation-1",
    }
}

result = graph.invoke(input_state, config)
```

### Checkpointer

Checkpointer는 그래프의 State를 실행 흐름별로 저장하고 다시 불러오는 인터페이스다. `thread_id`를 기본 식별자로 사용해 각 Thread의 Checkpoint를 구분한다.

| 구현체 | 주요 용도 |
| --- | --- |
| `InMemorySaver` | 같은 프로세스 안에서 진행하는 학습과 테스트 |
| SQLite 기반 Saver | 로컬 애플리케이션과 가벼운 영속 저장 |
| PostgreSQL 기반 Saver | 운영 환경의 지속적인 상태 저장 |

`InMemorySaver`를 사용하면 같은 `thread_id`로 여러 번 `invoke()`했을 때 이전 State를 이어 갈 수 있다. 다만 메모리에 저장되므로 프로세스가 종료되면 상태도 사라진다.

### Thread

Thread는 같은 `thread_id`로 묶인 여러 번의 그래프 실행을 하나의 흐름으로 관리하는 실행 세션 단위다.

같은 그래프를 사용하더라도 서로 다른 `thread_id`를 전달하면 상태가 섞이지 않는 독립적인 실행 흐름이 만들어진다. 따라서 사용자, 대화 또는 작업마다 고유한 값을 지정해야 한다.

```python
config = {
    "configurable": {
        "thread_id": "user-42-session-1",
    }
}
```

### Checkpoint

Checkpoint는 **특정 super-step이 끝난 시점의 그래프 State와 실행 메타데이터를 저장한 스냅숏**이다.

Super-step은 해당 단계에 예정된 하나 이상의 노드가 실행되는 그래프의 단위 시점이다. 병렬 노드가 있다면 같은 super-step 안에서 함께 실행될 수 있으며, 완료된 결과가 합쳐진 뒤 Checkpoint가 만들어진다.

저장된 상태는 다음 API로 확인하거나 갱신할 수 있다.

- `get_state(config)`: Thread의 최신 State 조회
- `get_state_history(config)`: Thread의 Checkpoint 이력 조회
- `update_state(config, values)`: 기존 상태를 직접 수정하는 대신, 수정된 값을 반영한 새 Checkpoint 생성

### Durable Execution

Durable Execution은 그래프 실행이 중간에 중단되더라도 저장된 Checkpoint를 기준으로 작업을 이어 갈 수 있게 하는 실행 모델이다.

LLM 호출이나 외부 API 작업처럼 실행 시간이 길거나 실패 가능성이 있는 워크플로에서는 처음부터 모든 단계를 다시 수행하는 비용을 줄일 수 있다.

다만 Checkpoint가 있다고 해서 모든 부수 효과가 자동으로 안전해지는 것은 아니다. 재개 과정에서 작업이 다시 실행될 가능성을 고려해 다음 원칙을 함께 적용해야 한다.

- 비결정적인 작업과 외부 부수 효과를 별도의 Node 또는 `@task`로 분리한다.
- 데이터 저장이나 결제처럼 중복 실행이 위험한 작업은 멱등하게 설계한다.
- 운영 환경에서는 프로세스 종료 후에도 State가 남는 영속 Checkpointer를 사용한다.
- 재개할 때 기존 실행과 같은 `thread_id`를 사용한다.

### 전체 동작 흐름

```text
그래프 컴파일 시 Checkpointer 연결
        ↓
thread_id와 함께 그래프 실행
        ↓
각 super-step의 State를 Checkpoint로 저장
        ↓
실행 중단 또는 다음 요청 발생
        ↓
같은 thread_id의 Checkpoint 조회
        ↓
저장된 State를 바탕으로 실행 재개
```
