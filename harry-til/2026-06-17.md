# 📝 TIL (Today I Learned)

## [핵심 배운 내용]

- RAG 평가는 크게 **검색 품질(Retrieval Quality)**과 **생성 품질(Generation Quality)**을 나누어 본다.
- 검색 품질은 `Context Precision`, `Context Recall`로 확인하고, 생성 품질은 `Faithfulness`, `Answer Relevancy`로 확인한다.
- `Faithfulness`는 답변이 검색된 문맥에 실제로 근거하는지를 확인하므로, RAG에서 hallucination을 점검하는 핵심 지표다.
- `LLM-as-a-Judge`는 LLM에게 채점 기준과 평가 대상을 제공해 답변 품질을 자동으로 평가하게 하는 방식이다.
- RAG에서 indexing과 retrieval은 보통 모델의 가중치를 바꾸는 학습이 아니라, 모델이 참고할 context를 구성하는 과정으로 이해해야 한다.

## [개념 정리]

### RAG 평가

RAG 평가는 RAG 파이프라인의 검색 품질과 생성 품질을 정량적 지표로 측정하는 과정이다.

| 평가 대상 | 지표 | 측정 내용 | 핵심 질문 |
| --- | --- | --- | --- |
| 검색 품질(Retrieval) | Context Precision | 검색된 문맥 중 관련 있는 문맥의 비율 | 검색 결과에 쓸모없는 문서가 섞여 있지는 않은가? |
| 검색 품질(Retrieval) | Context Recall | 정답에 필요한 정보 중 검색된 비율 | 정답에 필요한 정보를 빠뜨리지 않았는가? |
| 생성 품질(Generation) | Faithfulness | 답변이 검색된 문맥에 충실한 정도 | LLM이 문맥에 없는 내용을 지어내지 않았는가? |
| 생성 품질(Generation) | Answer Relevancy | 답변이 질문에 관련된 정도 | 답변이 사용자 질문의 의도에 맞는가? |

### 검색 품질 지표

#### Context Precision

- 검색 결과로 가져온 문서 중 실제로 질문에 도움이 되는 문서가 얼마나 많은지 평가한다.
- 검색된 문맥이 얼마나 깔끔하고 순도가 높은지를 보는 지표다.
- 값이 낮다면 관련 없는 문서가 검색 결과에 많이 섞였을 가능성이 있다.

#### Context Recall

- 질문에 답하는 데 꼭 필요한 정보가 검색 결과 안에 포함되어 있는지 평가한다.
- 검색 결과가 충분한지, 필요한 근거를 빠뜨리지 않았는지를 보는 지표다.
- 값이 낮다면 답변에 필요한 핵심 문서를 검색하지 못했을 가능성이 있다.

### 생성 품질 지표

#### Faithfulness

- 생성된 답변이 검색된 문맥에 얼마나 충실한지 평가한다.
- 단순히 “답변이 맞아 보이는가”가 아니라, 검색된 문서 안에 실제로 있는 내용만 말했는지를 본다.
- RAG에서 hallucination을 직접적으로 확인하는 중요한 지표다.

#### Answer Relevancy

- 답변이 사용자 질문의 의도에 맞는지 평가한다.
- 답변이 문맥에는 충실하더라도, 사용자가 물은 핵심을 제대로 답하지 못할 수 있다.
- 따라서 Faithfulness와 Answer Relevancy는 함께 확인해야 한다.

### RAG 평가 지표 요약

- **Context Precision**: 검색 결과에 noise가 얼마나 적은가?
- **Context Recall**: 필요한 근거를 얼마나 빠짐없이 찾았는가?
- **Faithfulness**: 답변이 문맥에 없는 내용을 지어내지 않았는가?
- **Answer Relevancy**: 답변이 질문 의도에 정확히 맞았는가?

정리하면 RAG 평가는 “충분한 자료를 얼마나 순도 있게 검색했는가”와 “질문 의도에 맞게 신뢰성 있는 답을 생성했는가”를 함께 보는 과정이다.

### 평가 데이터셋 구성

| 구성 요소 | 설명 | 예시 | 생성자 |
| --- | --- | --- | --- |
| `question` | 사용자 질문 | `"어댑터즈는 무엇인가요?"` | 사람 |
| `ground_truth` | 사람이 작성한 정답 | `"어댑터즈는 스타트업코드에서 제공하는 개발 책 서빙 서비스입니다."` | 사람 |
| `contexts` | Retrieval이 반환한 문맥 | `["어댑터즈는 스타트업코드에서...", "다양한 개발 분야의..."]` | RAG 파이프라인 |
| `answer` | Generation이 생성한 답변 | `"어댑터즈는 개발 책 서빙 서비스입니다."` | RAG 파이프라인 |

### 평가 결과 해석 매트릭스

| 상황 | Context Precision | Context Recall | Faithfulness | Answer Relevancy | 진단 | 개선 방향 |
| --- | --- | --- | --- | --- | --- | --- |
| 모두 양호 | 높음 | 높음 | 높음 | 높음 | 정상 | 현재 상태 유지 |
| 검색 실패 | 낮음 | 낮음 | 낮음 | 낮음 | 필요한 문서를 찾지 못함 | 임베딩 모델 변경, 청킹 전략 개선, Hybrid Search 도입 |
| 검색 노이즈 | 낮음 | 높음 | 낮음 | 낮음 | 관련 문서는 있지만 무관 문서도 많음 | Reranking 도입, Metadata Filtering 적용 |
| 생성 실패 | 높음 | 높음 | 낮음 | 낮음 | 문서는 잘 찾았지만 LLM이 잘못 생성 | 프롬프트 개선, temperature 조정, 모델 변경 |
| 답변 이탈 | 높음 | 높음 | 높음 | 낮음 | 문맥에 충실하지만 질문 의도와 다름 | 프롬프트에 질문 재확인 지시 추가 |

### LLM-as-a-Judge

LLM-as-a-Judge는 LLM에게 채점 프롬프트를 제공해 RAG 평가 지표를 자동으로 판단하게 하는 평가 기법이다.

- 채점 기준과 평가 대상을 프롬프트에 포함한다.
- LLM의 판단 결과를 점수나 Yes/No 형식으로 받는다.
- 응답을 파싱해 정량 점수로 변환한다.

| 순서 | 단계 | 설명 | 예시 |
| --- | --- | --- | --- |
| 1 | 채점 프롬프트 작성 | 채점 기준과 평가 대상인 질문, 문맥, 답변을 프롬프트에 포함한다. | `"다음 답변이 문맥에 의해 뒷받침되는지 0~1 점수로 판단하세요."` |
| 2 | LLM에게 전달 | 채점 프롬프트를 LLM API에 전달한다. | Gemini API, OpenAI API 등 |
| 3 | 응답 파싱 | LLM의 응답을 점수로 변환한다. | `"0.85"` → `float(0.85)` |

LLM-as-a-Judge에서 가장 중요한 단계는 채점 프롬프트 작성이다. 채점 기준이 구체적일수록 판단이 일관되고, 출력 형식이 단순할수록 파싱이 쉬워진다.

### RAG 평가 도구

- RAG 평가에는 `Ragas`, `DeepEval` 같은 도구를 활용할 수 있다.
- 이런 도구들은 retrieval과 generation 품질을 빠르게 반복 평가하는 데 도움을 준다.

## [더 알아볼 내용]

### 프롬프트 엔지니어링과 컨텍스트 엔지니어링

- **프롬프트 엔지니어링**은 생성형 AI로부터 원하는 출력을 얻기 위해 자연어 입력을 구조화하거나 보강하는 과정이다.
- **컨텍스트 엔지니어링**은 모델에게 제공할 정보의 범위와 흐름을 설계하는 영역이다.
- 시스템 지침, 메타데이터, API 도구, 검색된 문서, 대화 기록 등이 모두 context 설계 대상이 될 수 있다.
- RAG는 컨텍스트 엔지니어링의 대표적인 구현 방식 중 하나로 볼 수 있다.

### LLM 학습의 의미 구분

- **모델 학습**: 모델의 가중치가 실제로 업데이트되는 과정이다.
- **In-context Learning**: 프롬프트나 context 안의 정보를 바탕으로 일시적으로 더 나은 답을 하는 능력이다.
- In-context Learning은 임시적인 동작이며, 모델의 가중치가 바뀌는 것은 아니다.

### Hermes Agent와 Skill 문서화

복잡한 workflow를 성공적으로 수행한 뒤, 그 과정을 재사용 가능한 skill 문서로 저장하면 다음 세션에서 시행착오를 줄일 수 있다.

```text
[사용자 요청]
  → [Personal Agent]
  → [기존 스킬 매칭?]
      → 있으면 즉시 실행
      → 없고 복잡하면 Sub-Agent 실행
  → [작업 완수 및 피드백]
  → [새로운 절차를 Markdown Skill 문서로 저장]
  → [다음 세션에 재활용]
```

현재 TIL workflow의 `TIL_AUTOMATION_LOG.md`는 아직 정식 `SKILL.md`는 아니지만, 나중에 `til-publisher` skill로 승격하기 위한 재료를 쌓는 역할을 한다.

## [오늘의 회고]

- RAG를 잘 만든다는 것은 단순히 검색과 생성을 연결하는 것이 아니라, 검색 품질과 생성 품질을 각각 측정하고 개선할 수 있어야 한다는 뜻이다.
- 특히 RAG에서는 답변이 그럴듯한지보다, 검색된 문맥에 근거하고 있는지가 중요하다.
- 앞으로 프로젝트를 만들 때도 “잘 동작한다”에서 멈추지 말고, 어떤 지표로 무엇을 개선할지 함께 기록해야겠다.
