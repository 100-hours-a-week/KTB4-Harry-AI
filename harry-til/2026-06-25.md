## [개념 정리]

### RAG 베이스라인 구조 복습

RAG 베이스라인은 사용자의 질문에 답하기 위해 문서를 검색하고, 검색된 문맥을 LLM에 전달해 답변을 생성하는 기본 구조다.

```text
문서 로드
-> 문서 분할
-> 임베딩
-> Vector DB 저장
-> Retriever 생성
-> Prompt 구성
-> LLM 호출
-> 답변 반환
```

이 흐름에서 중요한 점은 인덱싱과 질의 응답을 분리하는 것이다.

문서를 Vector DB에 저장하는 인덱싱 과정은 비용이 크고 시간이 걸릴 수 있다. 따라서 FastAPI 요청이 들어올 때마다 문서를 다시 인덱싱하면 응답 속도와 운영 비용이 크게 나빠진다.

### FastAPI에서 RAG를 서빙할 때의 구조

FastAPI로 RAG를 서빙할 때는 보통 다음처럼 책임을 나눌 수 있다.

| 단계 | 역할 |
| --- | --- |
| 인덱싱 함수 | 문서를 로드하고 분할한 뒤 Vector DB에 저장 |
| chain 생성 함수 | Retriever, Prompt, LLM, Parser를 묶어 RAG chain 생성 |
| API endpoint | 사용자 요청을 받아 준비된 chain을 호출하고 응답 반환 |

요청마다 새로 해야 하는 일과, 애플리케이션 시작 시 한 번만 해도 되는 일을 구분하는 것이 핵심이다.

### FastAPI `lifespan`

FastAPI의 `lifespan`은 애플리케이션 시작과 종료 시점에 실행할 로직을 정의하는 방식이다.

RAG 서빙에서는 애플리케이션이 시작될 때 다음과 같은 작업을 수행할 수 있다.

- 환경변수 로드
- Vector DB 또는 Retriever 준비
- RAG chain 생성
- 전역 상태나 `app.state`에 chain 저장

```python
app = FastAPI(lifespan=lifespan)
```

이렇게 초기화해 두면 각 요청에서는 이미 준비된 chain을 사용해 답변을 생성할 수 있다.

### 환경변수와 의존성 관리

`.env` 파일은 API key, provider 설정처럼 공개되면 안 되는 값을 담는 데 사용한다.

공개 repo에서는 `.env`를 반드시 `.gitignore`에 포함해야 한다. 대신 필요한 환경변수 이름만 보여주는 `.env.example` 같은 파일을 둘 수 있다.

오늘 확인한 의존성 관리 포인트:

- `dotenv` 대신 `python-dotenv`를 사용한다.
- `pyproject.toml`을 직접 수정한 뒤에는 환경에 바로 반영되지 않을 수 있으므로 `uv sync`가 필요하다.
- 패키지를 교체할 때는 다음 흐름으로 정리할 수 있다.

```bash
uv remove dotenv
uv add python-dotenv
```

임시로 환경변수를 바꿔 실행하고 싶다면 터미널에서 다음처럼 설정할 수 있다.

```bash
export LLM_PROVIDER=google
```

### GitHub 공개 전 점검

RAG 예제나 FastAPI 프로젝트를 GitHub에 공개할 때는 코드뿐 아니라 포함되는 파일도 확인해야 한다.

특히 다음 파일은 주의가 필요하다.

- `.env`: API key 등 민감 정보가 들어갈 수 있으므로 공개하면 안 된다.
- 실습용 sample 파일: 실제로 필요하지 않다면 공개 전에 정리한다.
- 불필요한 PDF, HTML, 테스트용 데이터: repo 목적에 맞는지 확인한다.

Git 상태에서 `U`는 보통 untracked, 즉 아직 Git이 추적하지 않는 파일을 의미한다. `git add` 후에는 staged 상태가 되며, 새 파일은 added 상태로 커밋 대상에 포함된다.

초기 코드 공개 시에는 프로젝트의 첫 상태를 담는 커밋 메시지를 명확하게 작성하는 것이 좋다.

```text
Drop initial code
```
