## [핵심 배운 내용]

- LangSmith는 LangChain 기반 LLM 애플리케이션의 실행 과정을 추적하고, 응답 품질을 평가하며, 프롬프트를 관리하기 위한 운영·평가 플랫폼이다.
- LLM 애플리케이션은 단순히 최종 답변만 보는 것보다, 각 단계의 입력·출력·지연 시간·토큰 사용량을 함께 관찰해야 개선 방향을 찾기 쉽다.
- Tracing은 RAG chain 내부에서 어떤 단계가 어떤 입력을 받고 어떤 출력을 냈는지 Run 단위로 기록하는 기능이다.
- Dataset과 Evaluator를 사용하면 프롬프트나 chain 변경이 답변 품질에 어떤 영향을 주는지 비교하고 회귀 테스트할 수 있다.
- LangChain 문법에 맞춰 구성 요소를 분리해 두면, 나중에 모델이나 Retriever를 교체하면서 실험하기 쉬워진다.

## [개념 정리]

### LangSmith

LangSmith는 LangChain으로 만든 LLM 애플리케이션을 운영 단계에서 관찰하고 개선하기 위한 도구다.

일반적인 코드에서는 중간 값을 확인하기 위해 `print()`를 넣거나 로그를 직접 남긴다. 하지만 LLM chain은 프롬프트, 모델 호출, Retriever, parser, tool call처럼 여러 단계가 이어져 있기 때문에, 단순 출력만으로는 문제가 어디에서 생겼는지 파악하기 어렵다.

LangSmith는 이러한 실행 과정을 Run 단위로 기록하고, 각 단계의 입력·출력·지연 시간·토큰 사용량 등을 트리 구조로 보여준다.

LangSmith를 사용하는 이유:

- chain의 각 단계에서 어떤 입력과 출력이 오갔는지 확인할 수 있다.
- 특정 응답이 왜 나왔는지 추적하기 쉽다.
- 비용, latency, token 사용량을 함께 볼 수 있다.
- 프롬프트나 chain 변경 전후의 품질을 비교할 수 있다.
- 평가 데이터셋을 기준으로 회귀 테스트를 수행할 수 있다.

### LangSmith 실습용 RAG 베이스라인 만들기

LangSmith를 활용하려면 먼저 관찰하고 평가할 기준이 되는 RAG chain이 필요하다.

RAG 베이스라인은 “일단 동작하는 최소 RAG 흐름”이라고 볼 수 있다.

```text
질문
-> Retriever로 관련 문서 검색
-> 검색된 문서를 prompt에 주입
-> LLM이 답변 생성
-> 답변 반환
```

이 베이스라인이 있어야 LangSmith에서 다음을 확인할 수 있다.

- Retriever가 어떤 문서를 가져왔는가?
- prompt에는 어떤 context가 들어갔는가?
- LLM은 어떤 답변을 생성했는가?
- 어느 단계에서 시간이 오래 걸렸는가?
- prompt나 Retriever 설정을 바꾸면 결과가 좋아지는가?

### Tracing

Tracing은 chain 실행 과정을 추적하는 기능이다.

하나의 사용자 요청이 들어왔을 때, LangSmith는 전체 실행을 하나의 Run으로 기록하고 그 안에 세부 단계를 계층적으로 보여준다.

예를 들어 RAG chain에서는 다음과 같은 단계가 trace에 남을 수 있다.

```text
사용자 질문
-> Retriever 호출
-> 검색된 Document 목록
-> Prompt 구성
-> LLM 호출
-> 최종 답변
```

Tracing의 핵심은 최종 답변만 보는 것이 아니라, 답변이 만들어지기까지의 흐름을 함께 보는 것이다.

### Dataset

Dataset은 LangSmith에 등록된 입력과 기대 출력 쌍의 모음이다.

평가, 회귀 테스트, 프롬프트 비교의 기준이 되는 표준 자료로 사용할 수 있다.

예를 들어 RAG 평가용 Dataset은 다음과 같은 형태가 될 수 있다.

| 입력 | 기대 출력 또는 평가 기준 |
| --- | --- |
| 질문 | 정답, 근거 문서, 기대 답변 방향 |

Dataset을 만들어 두면 프롬프트나 Retriever 설정을 바꾼 뒤, 같은 질문 묶음에 대해 결과가 좋아졌는지 비교할 수 있다.

### Evaluator

Evaluator는 LLM 애플리케이션의 출력 품질을 평가하는 기준 또는 평가 함수다.

평가 방식은 목적에 따라 달라질 수 있다.

- 정답과 얼마나 일치하는지 평가
- 답변이 질문에 충실한지 평가
- 검색된 문서 근거에 기반해 답했는지 평가
- 형식 요구사항을 지켰는지 평가
- 이전 버전보다 결과가 좋아졌는지 비교

LLM 애플리케이션은 정답이 하나로 고정되지 않는 경우가 많기 때문에, Evaluator를 통해 품질을 일관된 기준으로 비교하는 것이 중요하다.

### Prompt Management

Prompt Management는 프롬프트를 버전 관리하고 실험하는 과정이다.

LLM 애플리케이션에서는 프롬프트 문장 하나만 바뀌어도 답변 품질이 크게 달라질 수 있다. 따라서 어떤 프롬프트를 사용했는지, 변경 전후 결과가 어떻게 달라졌는지 기록하는 것이 중요하다.

Prompt Management를 통해 얻을 수 있는 장점:

- 프롬프트 변경 이력을 추적할 수 있다.
- 여러 프롬프트 버전을 비교할 수 있다.
- 특정 실험에서 어떤 프롬프트가 사용됐는지 확인할 수 있다.
- 평가 결과와 연결해 더 나은 프롬프트를 선택할 수 있다.

### LangChain 구조로 감싸는 이유

RAG를 처음부터 직접 구현하면 자유도는 높지만, 관찰·교체·평가 구조를 직접 만들어야 한다.

반대로 LangChain 문법에 맞게 구성 요소를 나눠 두면 나중에 모델, Retriever, parser, prompt를 바꿔가며 실험하기 쉽다.

특히 직접 만든 LLM을 붙여보다가 성능이나 운영 문제가 있다면, 같은 인터페이스에 맞춰 사전학습 모델이나 외부 모델로 교체하는 전략을 생각해 볼 수 있다.

다만 구현 순서는 상황에 따라 달라질 수 있다.

- 모델 자체를 연구하거나 학습하는 것이 목표라면 모델 구현이 먼저일 수 있다.
- RAG 서비스 흐름을 빠르게 검증하는 것이 목표라면 LangChain 기반 RAG 구성을 먼저 만드는 편이 나을 수 있다.
