## [핵심 배운 내용]

- LangChain은 LLM 애플리케이션을 만들 때 필요한 기능을 컴포넌트 단위로 나누고, 이를 조합해 실행 흐름을 구성할 수 있게 해 주는 프레임워크다.
- LangChain의 `Message`는 LLM 제공자마다 다른 메시지 형식을 역할과 내용이 있는 구조화된 객체로 다루기 위한 추상화다.
- LCEL(LangChain Expression Language)은 `prompt | model | parser`처럼 Runnable 컴포넌트를 파이프(`|`) 연산자로 연결해 하나의 chain을 만드는 방식이다.
- 프롬프트 템플릿은 프롬프트를 문자열 조합이 아니라 재사용 가능한 객체로 관리하게 해 준다.
- Structured Output은 LLM 응답을 사람이 읽는 자연어가 아니라, 프로그램이 바로 사용할 수 있는 정해진 구조로 받기 위한 방식이다.

## [개념 정리]

### LangChain

LangChain은 LLM 애플리케이션에 필요한 기능을 여러 컴포넌트로 나누고, 이를 조합해서 하나의 실행 흐름으로 만들 수 있게 해 주는 프레임워크다.

LangChain을 사용하는 이유:

- 여러 LLM 모델을 비교하거나 교체하기 쉽다.
- 프롬프트를 객체로 관리해 재사용성과 유지보수성을 높일 수 있다.
- 메시지, 프롬프트, 모델, 파서, 검색, 메모리 같은 구성 요소를 일관된 방식으로 연결할 수 있다.
- LangSmith와 연동하면 chain 실행 과정의 입력, 출력, 중간 단계를 추적할 수 있어 디버깅에 도움이 된다.

### Message

`Message`는 Chat Model과 주고받는 내용을 역할과 함께 구조화한 객체다.

예를 들어 사용자 메시지, 시스템 메시지, AI 응답 메시지를 단순 문자열이 아니라 역할이 포함된 데이터 구조로 다룬다.

```python
from langchain_core.messages import HumanMessage, SystemMessage
```

Message를 사용하는 이유:

- 대화 맥락과 역할 정보를 구조적으로 전달할 수 있다.
- 멀티턴 대화에서 이전 대화 흐름을 유지하기 쉽다.
- LLM 제공자마다 다른 메시지 형식을 LangChain 내부에서 일관된 방식으로 다룰 수 있다.
- 도구 호출이나 모델 교체 같은 확장에도 같은 메시지 구조를 활용할 수 있다.

### LCEL(LangChain Expression Language)

LCEL은 LangChain 컴포넌트를 파이프(`|`) 연산자로 연결해 하나의 실행 흐름을 만드는 방식이다.

```python
chain = prompt | model | parser
```

이 구조에서 각 컴포넌트는 `Runnable` 인터페이스를 구현하고 있기 때문에 서로 연결할 수 있다.

`Runnable`은 다음과 같은 공통 실행 메서드를 제공한다.

| 메서드 | 의미 |
| --- | --- |
| `invoke()` | 하나의 입력을 실행하고 하나의 결과를 반환 |
| `stream()` | 결과를 스트리밍 형태로 반환 |
| `batch()` | 여러 입력을 한 번에 처리 |

LCEL을 사용하면 프롬프트 생성, 모델 호출, 출력 파싱을 하나의 chain으로 묶을 수 있다. 또한 LangSmith와 연동했을 때 각 단계의 입력과 출력이 추적되어, 문제가 발생했을 때 어느 단계에서 원인이 생겼는지 확인하기 쉽다.

### PromptTemplate

`PromptTemplate`은 변수 자리표시자를 포함한 프롬프트 문자열을 객체로 관리하고, 실행 시점에 값을 주입해 완성된 프롬프트를 생성하는 클래스다.

사용 이유:

- 프롬프트를 재사용 가능한 형태로 관리할 수 있다.
- 변수만 바꿔 여러 입력에 같은 프롬프트 구조를 적용할 수 있다.
- 프롬프트 수정과 실험을 반복할 때 유지보수 부담을 줄일 수 있다.

### ChatPromptTemplate

`ChatPromptTemplate`은 사용자와 LLM 사이의 메시지 기반 대화를 구성하기 위한 프롬프트 템플릿이다.

단순 문자열 프롬프트보다 대화 역할을 명확히 표현할 수 있기 때문에, Chat Model과 함께 사용할 때 자연스럽다.

### FewShotPromptTemplate

`FewShotPromptTemplate`은 소수의 입력과 출력 예시를 프롬프트에 포함해, LLM이 원하는 형식과 패턴을 따라 응답하도록 유도하는 템플릿이다.

예시가 필요한 경우:

- 응답 형식을 일정하게 만들고 싶을 때
- 모델에게 작업 패턴을 보여주고 싶을 때
- 설명만으로 원하는 결과가 잘 나오지 않을 때

### MessagesPlaceholder

`MessagesPlaceholder`는 `ChatPromptTemplate` 안에서 실행 시점에 메시지 리스트가 들어갈 자리를 지정하는 컴포넌트다.

대화 기록처럼 요청마다 달라지는 메시지 흐름을 역할 구조 그대로 주입할 수 있어, 멀티턴 대화에서 맥락을 유지하는 데 유용하다.

### Structured Output

Structured Output은 LLM 응답을 사전에 정의한 스키마에 맞춰 구조화된 객체로 받는 방식이다.

LLM의 자연어 응답은 사람이 읽기에는 편하지만, 프로그램이 바로 사용하기에는 형식이 흔들릴 수 있다. 같은 요청에도 어떤 때는 번호 목록, 어떤 때는 쉼표 구분, 어떤 때는 문장형 답변이 나올 수 있기 때문이다.

Structured Output은 이런 문제를 줄이기 위해 응답을 처음부터 정해진 데이터 구조로 받도록 만든다.

### JsonOutputParser

`JsonOutputParser`는 LLM의 텍스트 응답을 JSON 또는 Python 딕셔너리 형태로 파싱하는 출력 파서다.

특징:

- 프롬프트 지시를 통해 JSON 형식 출력을 유도한다.
- 모델 응답을 파싱해 프로그램에서 다루기 쉬운 구조로 바꾼다.
- 모델이 provider-native structured output을 직접 지원하지 않는 경우 대안이 될 수 있다.

### PydanticOutputParser

`PydanticOutputParser`는 LLM의 텍스트 응답을 Pydantic 모델 인스턴스로 파싱하는 출력 파서다.

Pydantic은 Python에서 데이터 구조를 정의하고 타입을 검증하는 라이브러리다. 이 파서를 사용하면 LLM 응답을 단순 JSON이 아니라 타입 검증이 가능한 객체로 다룰 수 있다.

관련 메서드:

- `get_format_instructions()`: 모델이 따라야 할 JSON 스키마 지시문을 제공한다.
- 파싱 결과는 Pydantic 모델로 변환되어 `result.title`처럼 필드 단위로 접근할 수 있다.

## [오늘의 회고]

오늘은 LangChain을 단순히 LLM을 호출하기 위한 wrapper가 아니라, LLM 애플리케이션의 흐름을 컴포넌트 단위로 구성하고 추적하기 위한 프레임워크로 이해하는 것이 중요했다.

특히 `prompt | model | parser` 구조를 보면 프롬프트 작성, 모델 호출, 출력 파싱이 각각 분리되면서도 하나의 실행 흐름으로 연결된다는 점이 명확해진다. 앞으로 LangChain을 볼 때는 개별 클래스의 사용법뿐 아니라, 이 컴포넌트들이 어떤 실행 흐름 안에서 연결되는지 함께 봐야겠다.
