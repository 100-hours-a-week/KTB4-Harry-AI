## [핵심 배운 내용]

- RAG에서 긴 문서는 그대로 LLM에 넣기 어렵기 때문에, Text Splitter로 적절한 크기의 chunk로 나누는 과정이 필요하다.
- Embeddings는 텍스트를 벡터로 변환하는 추상화이고, VectorStore는 이 벡터를 저장하고 유사도 검색을 수행하는 저장소 인터페이스다.
- Retriever는 “질문을 입력받아 관련 `Document` 리스트를 반환한다”는 검색 컴포넌트의 입출력 규약을 표준화한다.
- 고급 Retriever는 단순 유사도 검색의 한계를 보완하기 위해 질의 확장, 결과 압축, parent-child chunk 전략 등을 사용한다.
- RAG 검색 품질은 chunk 크기, overlap, 임베딩 모델, VectorStore, Retriever 전략이 함께 영향을 준다.

## [개념 정리]

### Text Splitter

Text Splitter는 긴 텍스트나 `Document` 리스트를 더 작은 `Document` 조각으로 나누는 LangChain 컴포넌트다.

Document Loader가 파일, PDF, 웹 페이지 등을 읽어 `Document` 객체로 변환하면, Text Splitter는 이 문서를 LLM과 검색에 적합한 크기의 chunk로 분할한다.

Text Splitter를 사용하는 이유:

- 긴 문서 전체를 LLM에 전달하기 어렵기 때문이다.
- 필요한 근거가 포함된 chunk만 검색해 답변 품질을 높일 수 있다.
- LLM에 전달되는 context 크기를 제한해 비용과 latency를 관리할 수 있다.
- chunk 크기와 overlap을 조정해 검색 정확도와 문맥 보존 사이의 균형을 잡을 수 있다.

### CharacterTextSplitter

`CharacterTextSplitter`는 하나의 구분자를 기준으로 텍스트를 나누는 기본 Text Splitter 구현체다.

예를 들어 다음과 같이 구분자를 설정할 수 있다.

| 설정 | 기준 |
| --- | --- |
| `separator="\n\n"` | 빈 줄 기준 분할 |
| `separator="\n"` | 줄바꿈 기준 분할 |
| `separator="---"` | `---` 문자열 기준 분할 |

구분자로 텍스트를 먼저 나눈 뒤, 앞에서부터 조각을 합쳐 `chunk_size`에 맞는 chunk를 만든다. `chunk_overlap`을 설정하면 인접한 chunk 사이에 일부 텍스트가 겹쳐 들어간다.

### RecursiveCharacterTextSplitter

`RecursiveCharacterTextSplitter`는 구분자 리스트를 우선순위대로 시도하면서 텍스트를 나누는 구현체다.

먼저 큰 의미 경계를 기준으로 나누고, `chunk_size`를 초과하는 조각이 있으면 더 작은 구분자로 다시 나누는 과정을 반복한다. 이 방식은 가능한 한 자연스러운 의미 경계를 유지하면서 chunk 크기를 통제하는 데 유용하다.

### TokenTextSplitter

`TokenTextSplitter`는 텍스트를 토큰으로 변환한 뒤, 토큰 수 기준으로 chunk를 나누는 Text Splitter다.

글자 수와 토큰 수는 항상 비례하지 않는다. 특히 한국어처럼 한 글자나 짧은 표현이 여러 토큰으로 나뉠 수 있는 언어에서는, 실제 LLM 입력 한도를 맞추려면 글자 수보다 토큰 수 기준 분할이 더 정확할 수 있다.

LangChain의 `TokenTextSplitter`는 OpenAI의 `tiktoken` 기반 토크나이저를 사용할 수 있다.

### Embeddings

LangChain에서 `Embeddings`는 텍스트를 벡터로 변환하는 추상 인터페이스다.

RAG에서는 문서와 사용자 질의를 같은 벡터 공간에 매핑한 뒤, 두 벡터의 유사도를 기준으로 관련 문서를 검색한다.

| 메서드 | 입력 | 출력 | 사용 시점 |
| --- | --- | --- | --- |
| `embed_documents(texts)` | `List[str]` | `List[List[float]]` | 문서 인덱싱 |
| `embed_query(text)` | `str` | `List[float]` | 사용자 질의 검색 |

`Embeddings` 추상화를 사용하면 임베딩 모델 종류와 무관하게 같은 방식으로 문서 벡터화와 질의 벡터화를 수행할 수 있다.

### VectorStore Interface

VectorStore는 임베딩 벡터를 저장하고 검색하는 저장소 인터페이스다.

LangChain의 VectorStore 인터페이스를 사용하면 Chroma, FAISS, Pinecone 같은 서로 다른 벡터 저장소를 비슷한 방식으로 다룰 수 있다.

대표적인 흐름:

```python
vectorstore = Chroma.from_documents(documents, embedding)
docs = vectorstore.similarity_search(query)
retriever = vectorstore.as_retriever()
```

VectorStore 인터페이스를 사용하는 이유:

- 벡터 DB 종류를 바꿔도 인덱싱과 검색 코드를 크게 바꾸지 않아도 된다.
- `from_documents()`로 문서를 인덱싱할 수 있다.
- `similarity_search()`로 유사도 검색을 수행할 수 있다.
- 필요하면 `as_retriever()`로 Retriever 형태로 변환해 LCEL chain에 연결할 수 있다.

### Chroma

Chroma는 임베딩 벡터를 로컬 파일 또는 인메모리 모드로 저장하고 검색할 수 있는 오픈소스 벡터 데이터베이스다.

별도의 데이터베이스 서버를 띄우지 않고 Python 프로세스 안에서 라이브러리처럼 사용할 수 있어, RAG 구조를 처음 실습하거나 작은 데이터셋으로 검색 품질을 빠르게 확인할 때 적합하다.

### FAISS

FAISS는 Meta가 공개한 고속 유사도 검색 라이브러리다.

로컬 메모리에서 대량 벡터 검색을 빠르게 수행할 수 있어, 외부 서버 호출 없이 RAG 검색 속도와 처리량을 높이고 싶을 때 사용할 수 있다.

### Retriever

Retriever는 질의를 입력받아 관련 `Document` 리스트를 반환하는 검색용 인터페이스다.

핵심은 “어떤 방식으로 검색하느냐”보다 검색 컴포넌트가 어떤 입력과 출력을 가져야 하는지 표준화한다는 점이다.

```text
input:  str
output: List[Document]
```

Retriever는 LCEL chain에 연결할 수 있는 Runnable로 다룰 수 있고, LangChain에서는 `BaseRetriever`를 기준으로 다양한 Retriever가 구현된다.

### Ensemble Retriever

`EnsembleRetriever`는 여러 Retriever가 반환한 검색 결과를 하나로 통합하는 합성 Retriever다.

예를 들어 dense vector 기반 의미 검색과 sparse vector 기반 키워드 검색을 함께 사용해 하이브리드 검색을 구성할 수 있다.

결과 통합에는 RRF(Reciprocal Rank Fusion) 같은 방식이 사용될 수 있다. RRF는 각 검색 결과의 순위를 기반으로 점수를 계산해 여러 검색기의 결과를 하나의 순위로 합치는 방법이다.

### Multi-Query Retriever

`MultiQueryRetriever`는 사용자의 원본 질의를 LLM으로 여러 변형 질의로 확장하고, 각 질의의 검색 결과를 통합하는 고급 Retriever다.

같은 의도라도 사용자가 어떤 표현을 쓰느냐에 따라 검색 결과가 달라질 수 있다. Multi-Query Retriever는 이 문제를 줄이기 위해 원본 질의를 여러 관점의 검색 질의로 바꿔 검색 범위를 넓힌다.

흐름:

```text
사용자 질의
-> LLM이 여러 변형 질의 생성
-> 각 질의를 base retriever에 전달
-> 검색 결과 통합
-> Document 리스트 반환
```

### Contextual Compression

Contextual Compression은 기본 Retriever가 가져온 `Document` 중에서 질의와 관련된 내용만 남기거나, 관련성이 낮은 문서를 제거하는 방식이다.

검색된 chunk가 길거나 불필요한 내용을 많이 포함하면 LLM에 전달되는 context가 비효율적일 수 있다. 이때 Contextual Compression을 사용하면 프롬프트 토큰을 줄이고 답변 품질을 안정화할 수 있다.

`ContextualCompressionRetriever`는 보통 두 구성 요소를 함께 가진다.

| 구성 요소 | 역할 |
| --- | --- |
| `base_retriever` | 후보 `Document` 리스트를 먼저 검색 |
| `base_compressor` | 검색된 `Document`에서 질의와 관련된 내용만 남기거나 관련 없는 문서를 제거 |

압축기 예시:

- `LLMChainExtractor`: 문서 안에서 질의와 관련된 문장만 추출
- `LLMChainFilter`: 문서 전체가 질의와 관련 있는지 판단
- `EmbeddingsFilter`: 임베딩 유사도 기준으로 문서를 필터링
- `DocumentCompressorPipeline`: 여러 압축기를 순서대로 적용

### Parent Document Retriever

`ParentDocumentRetriever`는 작은 child chunk로 검색하되, 최종 반환 시에는 해당 child chunk가 포함된 더 큰 parent document를 반환하는 Retriever다.

작은 chunk는 정확한 의미 검색에 유리하고, 큰 parent document는 LLM에게 더 넓은 문맥을 제공하는 데 유리하다.

| 단위 | 역할 |
| --- | --- |
| child chunk | 정확한 검색을 위한 작은 조각 |
| parent document | LLM에게 더 넓은 문맥을 전달하기 위한 큰 문서 |

`ParentDocumentRetriever`는 VectorStore에 child chunk를 저장하고, 별도의 docstore에 parent document를 저장한다. parent와 child의 크기를 독립적으로 조정할 수 있기 때문에 검색 정확도와 문맥 보존을 함께 고려할 수 있다.
