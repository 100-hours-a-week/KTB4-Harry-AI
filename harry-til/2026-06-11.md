- Transformer에서 **마스킹(Masking)**은 특정 위치의 정보를 attention 계산에서 참조하지 못하도록 막는 기법이다.

- **Layer Normalization**은 각 토큰 벡터의 분포를 정규화해, Transformer의 깊은 층에서도 학습이 안정적으로 이어지도록 만든다.
- AI 도구를 사용할 때도 핵심은 결과물의 양이 아니라, 문제를 어떻게 나누고 검증하며 문서화하는지에 있다.

## [개념 정리]

### 마스킹(Masking)

- Masking은 attention 연산에서 특정 위치의 정보를 참조하지 못하도록 차단하는 기법이다.
- Transformer에서는 주로 **Padding Mask**와 **Look-ahead Mask(Causal Mask)**를 사용한다.

#### Padding Mask

- 시퀀스 길이를 맞추기 위해 추가한 `PAD` 토큰을 attention 계산에서 무시하기 위해 사용한다.
- Encoder와 Decoder 모두에서 사용할 수 있다.
- `PAD` 토큰은 실제 의미를 가진 토큰이 아니므로, 모델이 이 위치에 attention을 주면 문맥 계산이 왜곡될 수 있다.

#### Look-ahead Mask / Causal Mask

- Decoder가 현재 위치 이후의 토큰을 참조하지 못하도록 막는 마스크다.
- 학습 중 정답 문장을 입력으로 사용하더라도, 모델이 미래의 정답 토큰을 미리 보지 못하게 한다.
- 보통 상삼각 행렬 형태로 미래 위치를 가린다.

```text
[ I | love | teaching | AI ]
  1     2        3        4
```

예를 들어 causal masking이 없다면:

- `"love"`를 예측할 때 `"teaching"`, `"AI"`를 미리 볼 수 있다.
- `"teaching"`을 예측할 때 `"AI"`를 이미 참고할 수 있다.

이렇게 되면 모델이 실제 생성 상황보다 쉬운 조건에서 학습하게 되므로, Decoder 학습에서는 미래 토큰을 가리는 causal masking이 필요하다.

### 잔차 연결(Residual Connection)

- 잔차 연결은 layer의 입력을 출력에 직접 더하는 구조다.

```text
output = x + Sublayer(x)
```

- 모델이 완전히 새로운 출력을 직접 만드는 대신, 입력 `x`에 더할 변화량인 `Sublayer(x)`를 학습하도록 만든다.
- 그래서 residual connection은 skip connection이라고도 부른다.

#### 잔차 연결이 필요한 이유

- 신경망이 깊어지면 역전파 과정에서 기울기가 여러 layer를 거치며 점점 작아질 수 있다.
- 기울기가 너무 작아지면 앞쪽 layer의 가중치가 거의 업데이트되지 않는다.
- 이 문제를 **기울기 소실(Vanishing Gradient)**이라고 한다.

#### 잔차 연결의 효과

- 입력 `x`가 그대로 더해지기 때문에, 역전파 시 기울기가 흐를 수 있는 경로가 추가된다.
- 한 경로는 sublayer를 통과하고, 다른 경로는 입력으로 직접 이어진다.
- 이 직접 경로 덕분에 깊은 네트워크에서도 기울기가 더 안정적으로 전달될 수 있다.
- 모델은 “완전히 새로운 표현”이 아니라 “기존 표현에 보탤 변화량”을 학습하면 되므로 최적화가 쉬워진다.

### 레이어 정규화(Layer Normalization)

- Layer Normalization은 각 샘플의 feature 차원을 기준으로 평균과 분산을 계산해 정규화하는 기법이다.
- Transformer에서는 residual connection 직후에 layer normalization을 적용하는 구조를 자주 사용한다.

```text
output = LayerNorm(x + Sublayer(x))
```

#### 레이어 정규화가 필요한 이유

- Attention과 FFN을 거치면서 각 token vector의 값 분포가 흔들릴 수 있다.
- Layer Normalization은 각 token vector의 분포를 평균 0, 분산 1에 가깝게 맞춰 다음 layer로 전달한다.
- 다음 layer는 비교적 일정한 분포의 입력을 받게 되므로, 가중치를 더 안정적으로 업데이트할 수 있다.

#### 레이어 정규화의 특징

- 각 token vector에 대해 평균과 분산을 구해 표준화한다.
- 이후 학습 가능한 `γ`, `β`를 적용해 필요한 분포로 다시 조정한다.
- batch 크기와 무관하게 동작하므로, 시퀀스 길이가 다양한 자연어 데이터에 잘 맞는다.
- 학습률을 더 크게 잡아도 학습이 폭주하지 않도록 돕는다.

## [더 알아본 내용]

### 좋은 소프트웨어 취향

- AI가 코드를 대량 생성하는 시대에는 단순한 구현 속도보다 **무엇을 만들지 판단하는 능력**이 더 중요해질 수 있다.
- 문제를 잘게 나누는 능력, 테스트와 신뢰성 설계, 확장성 판단, trade-off 감각이 엔지니어링의 핵심 역량으로 남는다.
- 참고: [좋은 소프트웨어 취향](https://news.hada.io/topic?id=30338)

### 하이퍼파라미터 설정과 기록

- Mini chatbot 예시에서 hyperparameter를 상단에 정의한 방식이 인상적이었다.
- 앞으로 모델을 만들 때는 hyperparameter를 “왜 이 값으로 설정했는지” 함께 기록해야 한다.
- 면접이나 포트폴리오에서도 “어떤 기준으로 값을 정했고, 어떤 실험을 통해 바꿨는지” 설명할 수 있어야 한다.

### Chatbot Architecture 아이디어

```text
사용자 입력
  → Front Agent
  → Backend Agent
  → Guardrail Agent
  → SSE Streaming
  → Client
```

- Client는 HTML/CSS/JavaScript 기반의 반응형 UI로 구성할 수 있다.
- 처음에는 휘발성 상태로 시작하더라도, 요구사항이 바뀌면 DB 저장 구조로 확장할 수 있다.
- 이미지 첨부가 있다면 최대 개수, OCR 처리 여부, 모든 이미지가 실제로 판독되는지 검증해야 한다.

### AI 활용 시 문서화 요청

- AI 도구로 코드를 생성할 때는 코드만 받는 것이 아니라, 함께 읽을 수 있는 명세서나 설명서도 요청하는 것이 좋다.
- 문서에는 다음 내용이 포함되면 좋다.
  - 전체 구조
  - 주요 파일과 역할
  - 핵심 함수/클래스 설명
  - 실행 방법
  - 변경하거나 확장할 때 봐야 할 지점
  - 테스트 또는 검증 방법

### 디버깅과 이해 방법

- 모델이나 파이프라인이 잘 이해되지 않을 때는 각 과정별 값을 `print`로 찍어보는 것이 가장 직관적일 수 있다.
- 입력이 어떻게 변환되고, 어떤 shape이나 값으로 다음 단계에 넘어가는지 직접 확인하면 구조 이해가 빨라진다.

## [오늘의 회고]

- AI를 사용해 결과물을 빠르게 만드는 것과 내 개발 실력이 실제로 늘어나는 것은 별개의 문제다.
- 도구를 쓰더라도 “내가 무엇을 설계했고, 무엇을 검증했고, 무엇을 배웠는가?”를 계속 점검해야 한다.
- 특히 챗봇처럼 AI 엔지니어링의 기초가 되는 작업은 API를 단순히 감싸는 수준을 넘어, 직접 구조를 만들고 성능을 개선해보는 경험이 중요하다.
- 앞으로는 모델 구조, 매개변수, 초매개변수, 실험 결과를 함께 조망하면서 기록하는 습관을 들이고 싶다.
- 작은 성능 개선이라도 꾸준히 시도하고 문서화하면, 나중에는 포트폴리오에서 그대로 설명할 수 있는 학습 자산이 될 수 있다.
