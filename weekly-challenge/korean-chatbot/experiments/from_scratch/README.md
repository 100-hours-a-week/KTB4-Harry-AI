# From-scratch 한국어 언어 모델 학습 실험

이 폴더는 토큰화부터 학습까지 직접 구현하며 언어 모델의 내부 흐름을
이해하기 위해 진행한 학습 실험을 보존한다.

## 현재 상태

- 상태: 일시 중단(Paused)
- 완료 범위: Step 01 토크나이저, Step 02 데이터셋, Step 03 작은 Transformer,
  Step 04 `calculate_loss()`
- 미완료 범위: 전체 학습 루프, auto-regressive 생성, FastAPI 서빙
- 중단 이유: 현재 학습 우선순위를 모델 내부 구현에서 RAG, LangChain,
  LangGraph, LangSmith 기반 애플리케이션 개발로 전환했다.

작은 seed corpus로 학습한 모델은 검색 문맥 이해, 지시 수행, 자연스러운 답변
생성 능력이 제한적일 것으로 예상한다. 코드는 실패한 구현이 아니라 학습 범위와
우선순위 변경 과정을 보여주는 실험 기록으로 남긴다. 이후 필요하면 프로젝트의
공통 Generator 경계에 연결해 사전학습 모델과 같은 입력으로 결과를 비교한다.

## 목표

auto-regressive 방식으로 문장을 생성할 수 있는 작은 from-scratch 한국어
언어 모델을 만든다.

1주차의 목표는 높은 모델 성능이 아니다. 목표는 학습 과정이 분명하게
남는 것이다.

1. 작은 한국어 텍스트를 준비한다.
2. 텍스트를 토큰과 id로 바꾼다.
3. id를 학습용 입력/정답 예제로 바꾼다.
4. 아주 작은 Transformer 언어 모델을 정의한다.
5. 몇 단계라도 직접 학습시켜 본다.
6. 다음 토큰을 반복해서 예측하며 문장을 생성한다.
7. 무엇이 바뀌었고, 무엇이 좋아졌고, 아직 무엇이 부족한지 기록한다.

## 페어 프로그래밍 규칙

각 단계는 한 번에 설명할 수 있을 정도로 작게 유지한다.

각 단계마다 다음 흐름을 따른다.

1. 이번 단계의 학습 목표를 말한다.
2. 코드를 실행하기 전에 어떤 결과가 나올지 먼저 예상한다.
3. 작은 함수나 클래스 하나만 구현한다.
4. 작은 검증을 하나 실행한다.
5. 배운 점을 두세 줄로 기록한다.

## 제안 구조

| 단계 | 파일 | 학습 목표 |
| --- | --- | --- |
| 01 | `src/step01_tokenizer.py` | 작은 한국어 텍스트를 준비하고, 토큰으로 나눈 뒤 id로 바꾼다. |
| 02 | `src/step02_dataset.py` | 다음 토큰 예측을 위한 입력/정답 예제를 만든다. |
| 03 | `src/step03_tiny_model.py` | 아주 작은 Transformer 언어 모델을 정의한다. |
| 04 | `src/step04_train_loop.py` | 모델을 학습시키고 loss 변화를 관찰한다. |
| 05 | `src/step05_generate.py` | 학습한 모델로 auto-regressive 문장 생성을 수행한다. |
| 06 | `src/step06_fastapi.py` | 생성 함수를 최소한의 FastAPI endpoint로 감싼다. |

이 구조는 튜토리얼과 실무 프로젝트의 중간 정도를 목표로 한다.

- `tokenizer`: 텍스트 전처리와 id 변환
- `dataset`: 학습 예제 생성
- `model`: 신경망 모델 정의
- `train_loop`: 학습 실행
- `generate`: 추론 및 생성 로직
- `fastapi`: 웹 API 경계

작은 말뭉치, vocabulary, 텐서 shape, 저장/불러오기, 실험 기록 같은 개념은
처음부터 별도 파일로 분리하지 않고, 필요한 단계 안에서 하나씩 다룬다.

## 기본 방향

처음에는 character-level 토큰화로 시작한다. 한국어 토큰화 품질 문제에
막히지 않고 전체 흐름을 이해하기 가장 쉽기 때문이다. 이후 regex 기반,
subword 기반, 학습형 토큰화로 바꾸는 과정을 개선 기록으로 남길 수 있다.

## 이 실험에서 제외한 것

- RAG
- LangChain
- LangGraph
- vector database
- 외부 hosted LLM API
- pretrained 챗봇 엔진

FastAPI는 원래 6단계로 계획했지만, 이 실험을 일시 중단하면서 구현하지 않았다.
