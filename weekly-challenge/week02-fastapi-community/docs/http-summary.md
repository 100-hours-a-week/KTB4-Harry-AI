# 📝 Week 02: HTTP 학습 요약 (Learned Summary)

## 1. HTTP란 무엇인가?

### 💡 정의 및 목적
* **HTTP (HyperText Protocol):** HTML, CSS, JS, 이미지 파일(PNG, JPEG) 및 구조화된 텍스트 데이터를 전송하기 위해 사용되는 **웹 표준 통신 규약**입니다.
* **사용 이유:** 웹 서버와 클라이언트가 매번 새로운 방식이 아닌, **표준화된 약속(규칙)**에 따라 안전하고 명확하게 하이퍼텍스트 문서를 주고받기 위함입니다.
* **사용 방법:** 클라이언트가 `URL`을 통해 특정 웹 서버 주소로 요청(Request)을 보냅니다.

### ✉️ HTTP Message 구조
클라이언트와 서버 간 데이터를 교환할 때 사용하는 메시지는 크게 3가지 파트로 구성됩니다.

| 구성 요소 | 설명 |
| :--- | :--- |
| **시작줄 (Start Line)** | 요청 시에는 `Method + URL + Version`, 응답 시에는 `Version + Status Code`가 포함되는 요약 정보 줄 |
| **헤더 (Headers)** | `Key: Value` 형태로 이루어진 편지 봉투. 본문 데이터의 타입, 크기, 인증 정보 등 부가적인 메타데이터를 포함 |
| **바디 (Body)** | 실제 전송하고자 하는 알맹이 데이터 (텍스트, JSON 데이터, 이미지 등) |

---

## 🛠️ Pydantic Model 기반 FastAPI의 Request Body 검증

FastAPI 백엔드 개발 시, **Pydantic Model**을 활용하면 HTTP Request Body(본문) 검증을 자동화할 수 있습니다.

* **자동 유효성 검사 (Validation):** 전달된 데이터를 Python 객체로 변환하는 과정에서 자동으로 검증을 수행합니다.
  * Request Body 데이터의 존재 여부
  * 필수 필드(Required Fields) 누락 여부
  * 각 필드의 데이터 타입 일치 여부
* **동작 시점:** 클라이언트의 요청이 라우터(Router) 함수에 도달하여 **실행되기 직전에 자동으로 검증을 수행**합니다.
* **장점:** 개발자가 컨트롤러단에서 데이터 검증을 위한 별도의 `if/else` 분기문이나 예외 처리 코드를 작성할 필요가 없어져 생산성이 극대화됩니다.

## 🚀 2. HTTP Request Method (요청 메서드)

* **정의:** 클라이언트가 서버에게 특정 리소스에 대해 **어떤 작업(행위)**을 수행하길 원하는지 명확하게 지시하는 명령 규칙입니다.
* **사용 방법:** HTTP 요청 메시지를 보낼 때 URL과 함께 메소드를 명시합니다.

| Method | 역할 | 핵심 특징 |
| :---: | :--- | :--- |
| **`GET`** | 리소스 **조회** | 서버의 데이터를 가져오기만 할 때 사용 (Body가 보통 비어있음) |
| **`POST`** | 리소스 **생성** | 서버로 데이터를 전송하여 새로운 리소스를 생성하거나 업데이트할 때 사용 |
| **`PUT`** | 리소스 **전체 수정** | 서버의 리소스를 전송된 데이터로 완전히 덮어씌워 업데이트할 때 사용 |
| **`PATCH`**| 리소스 **부분 수정** | 리소스의 전체가 아닌, 특정 필드(일부)만 수정 또는 업데이트할 때 사용 |
| **`DELETE`**| 리소스 **삭제** | 특정 리소스를 서버에서 삭제할 때 사용 |

## 🏁 3. HTTP Status Code (상태 코드)

* **정의:** 클라이언트의 HTTP 요청이 어떻게 처리되었는지 서버가 알려주는 **3자리 숫자 코드**입니다.
* **사용 방법:** 응답 메시지의 상태 줄(Status Line)에 포함되어 클라이언트에게 반환됩니다.

### 📊 상태 코드 분류 가이드
```text
1xx (정보 메시지) : 요청을 받았으며 작업을 계속 진행 중임을 의미
2xx (성공)       : 요청이 성공적으로 접수되고 처리됨 (예: 200 OK, 201 Created)
3xx (리다이렉션) : 브라우저가 요청한 URL 대신, 서버가 지정한 다른 URL로 자동 이동하도록 지시
4xx (클라이언트) : 클라이언트의 요청에 에러가 있음 (예: 400 Bad Request, 404 Not Found)
5xx (서버 오류)   : 서버 내부 문제로 요청을 처리할 수 없음 (예: 500 Internal Server Error)
```

## 📍 4. URL (Uniform Resource Locator)

* **정의:** 웹 상에 존재하는 특정 리소스(파일, 페이지 등)의 고유한 **위치와 종류를 나타내는 주소**입니다.

### 🔍 URL 구조 분석
> **예시 주소:** `http://www.example.kr:80/path/to/example.html?key1=value1&key2=value2`

| 구성 요소 | 예시 값 | 설명 |
| :--- | :--- | :--- |
| **Schema (프로토콜)** | `http` | 브라우저가 리소스를 요청할 때 사용해야 하는 통신 규약 |
| **Domain (호스트)** | `www.example.kr` | 리소스가 저장된 웹 서버의 주소 (영문 도메인 혹은 IP 주소) |
| **Port (포트 번호)** | `80` | 서버 내 리소스에 접근하기 위한 게이트웨이 (HTTP 기본 포트는 `80`, HTTPS는 `443`) |
| **Path (리소스 경로)** | `/path/to/example.html` | 웹 서버 내에서 실제 파일이 위치한 경로 (컴퓨터의 폴더 구조와 유사) |
| **Query Parameter** | `?key1=value1&key2=value2` | 웹 서버에 추가로 전달하는 파라미터(매개변수). `?`로 시작하며 `&`로 연결 |
